## 이번 세션 완료 (2026-06-22 — ★다기기 동기화 충돌 근본 수정: 리비전 기반 LWW) ✅사용자 확인됨

- **★★ 깃허브에서 포지션 종료해도 index 파일엔 포지션이 남던(되살아나던) 진짜 근본원인 수정** — 증상: 같은 계정인데 깃허브 사이트에서 종료하면 index 파일은 포지션 유지(+종료를 클라우드로 되돌림). **근본원인**: `onSnapshot`의 `if(localPosCount>0 && cloudPosCount===0){ doSyncSave(); }` 가드(=과거 '로그인 시 포지션 증발' 수정용 '로컬 보존'). "로컬엔 있고 클라우드엔 없음"만으로는 ①클라우드가 낡아 잘못 비었음(보존해야 함) vs ②다른 기기가 정상 종료함(버려야 함)을 **구분 불가** → 항상 ①로 오판해 열린 포지션을 클라우드에 재업로드 → 종료가 취소됨. **해결**: 상태에 단조증가 리비전 `S._rev` 도입(`bumpRev()`=`max(Date.now(), _rev+1)`, `save()`/`saveCritical()`에서만 증가·`loadState`에선 미증가). `onSnapshot`을 **리비전 LWW**로 교체 — 최초 스냅샷은 클라우드 로드(단 로컬 rev가 더 크면 보존, 둘 다 rev0 마이그레이션 과도기엔 포지션 보존), 이후 스냅샷은 `cloudRev>localRev→적용`/`<→로컬덮어씀`/`===→반향무시`. 옛 '로컬 보존'·'종료 보존'·`_lastSelfSaveTime 6초` 가드 제거(rev 동률이 곧 반향). `applyCloudData`가 클라우드 `_rev`를 로컬에 반영(적용 직후 되돌림 방지). 프리뷰 실측: ①깃허브종료→index 'APPLY_CLOUD(종료 반영)' ②반대방향 'KEEP_LOCAL' ③반향 'IGNORE' ④로그인 포지션 보존 'KEEP_LOCAL' ⑤초기로드 클라우드최신 'APPLY' ⑥⑦마이그레이션 보존/로드 — 7/7 통과, bumpRev 단조증가·applyCloudData rev반영·콘솔에러0.

## 이번 세션 완료 (2026-06-22 — 포지션 종료 후 페이지 재시작 시 포지션 부활 버그 외 6건 수정)

- **★ 포지션 종료 후 페이지 껐다 키면 포지션 부활하는 버그 근본 수정** — 근본원인: `closePosition()`이 디바운스된 `save()`를 호출했는데, `checkPending`이 매 1초마다 `save()→scheduleSyncSave()`를 호출해 2초 타이머를 계속 리셋 → 타이머가 영원히 발화 못함 → Firestore에는 종료 전 구버전(포지션 있는 상태) 유지. 리로드 시 `applyCloudData`가 구버전을 적용 → 포지션 부활. 해결: ①`saveCritical()` 함수 신설 — 디바운스 없이 즉시 Firestore에 저장(`clearTimeout`으로 기존 타이머 취소 후 `doSyncSave()` 직접 호출), `btcSimLastCritical` 타임스탬프도 저장. ②`closePosition`·`finishFill`·`checkLiquidation`에서 `save()` → `saveCritical()` 교체 → 포지션 변경 즉시 Firestore 반영. 프리뷰 실측: saveCritical_exists=true·closePosition_uses_saveCritical=true.
- **checkPending: 매 틱 save() 호출로 Firestore 타이머 영구 리셋되던 버그 수정** — 근본원인: `checkPending`이 미체결 주문이 있으면 매 1초마다 `save()`를 호출 → `scheduleSyncSave`가 2초 타이머를 매번 리셋 → TP/SL 주문 보유 중엔 Firestore에 데이터가 절대 저장되지 않음. 해결: `anyFilled` 플래그 추가, 지정가/TP/SL 체결 시만 `anyFilled=true` 설정, 루프 후 `if(anyFilled) save()`로 교체. 프리뷰 실측: 체결 없으면 save 0호출·체결 시에만 호출.
- **checkBgTpSl: 매 2초 save() 호출 + 항상 100% 청산하던 버그 수정** — 근본원인 2가지. ①체결 여부와 무관하게 매 2초마다 `save()` 호출 → 역시 Firestore 타이머 리셋. ②부분청산 TP/SL(`closeAmt`/`closeMode`)을 무시하고 항상 `delete S.positions[sym]`(100% 청산). 해결: ①`anyFilled` 플래그로 체결 시에만 save. ②`closeMode`/`closeAmt` 읽어 `frac` 계산(`usdt`/`coin`/`pct` 3가지), `closeQty=qty*frac`으로 실현손익 계산, `frac<1`이면 포지션 qty·margin 감소(부분청산), `frac≥1`이면 포지션 삭제 + 전체 TP/SL 제거. 프리뷰 실측: bg_only_saves_when_filled=true·bg_has_partial_close=true·bg_reads_closeMode_closeAmt=true.
- **onSnapshot 역방향 보호: 포지션 종료 직후 클라우드 구버전이 복원하는 케이스 차단** — 상황: `saveCritical()`로 즉시 저장하더라도 Firestore 전파 전에 리로드하면 `onSnapshot`이 구버전(포지션 있음)으로 응답 가능. 기존 보호(`_lastSelfSaveTime` 6초 가드)는 자기 저장 반향만 막고, 이 케이스는 무방비. 해결: `onSnapshot` 핸들러에 추가 분기 — 로컬 포지션 없음 + 클라우드 포지션 있음 + `btcSimLastCritical` 60초 이내일 때 → 로컬(종료 상태)이 올바름으로 판단하고 `doSyncSave()`로 클라우드 덮어씀. 프리뷰 실측: reverse_protection=true·uses_60s_window=true.
- **_firstSyncDone 가드: 앱 시작 직후 빈 상태로 Firestore 덮어쓰는 경쟁조건 수정** — 근본원인: 로그인 후 첫 `onSnapshot` 도달 전에 `onTick→checkPending→save()→doSyncSave`가 실행되면 로컬 빈 상태로 클라우드를 덮어씀. 해결: `_firstSyncDone` 플래그, `doSyncSave`에 `if(!_firstSyncDone) return` 가드(첫 스냅샷 전 업로드 차단), `startCloudListener`에서 스냅샷 수신 완료 후 `_firstSyncDone=true` 설정. 프리뷰 실측: doSyncSave_blocked_before_firstSync=true.
