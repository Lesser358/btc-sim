## 이번 세션 완료 (2026-06-25 — ★오더북 임밸런스: 호가 압력·벽 반영 [점진 고도화 3단계])

- **★ 신호 스캐너에 오더북 임밸런스(단기 호가 압력) 추가 — 스캘핑·단타 위주, 스윙 약하게, 중기 미적용** — 호가창의 매수/매도 대기 물량을 비교해 단기 압력을 점수에 반영. 호가는 조작(스푸핑) 가능해 보조 가중으로만 사용.
  - **`_aiObImbalance(depth,pct)`** — mid가 기준 ±pct 범위의 매수/매도 호가 **금액**을 합산해 `imb=(bid-ask)/(bid+ask)`(-1~1, +매수우위) 산출 + 벽 감지(`wall`: 한쪽 최대호가가 그 측 합의 50% 초과 & 반대측 최대의 1.5배 → 매수벽 +1/매도벽 −1). bids 내림차순·asks 오름차순 정렬 가정하고 범위 밖이면 break.
  - **데이터 조회** — `loadAiCoinSignals`에서 `cfg.wOb>0`일 때만 코인별 `fapi/v1/depth?limit=100` 추가 fetch(실패 시 null). `calcAiMarketSignalStyled(...,cfg,depth)` 시그니처에 depth 추가. **중기는 `wOb` 미설정이라 depth 조회 자체를 안 함**(불필요 네트워크 절약).
  - **점수 반영** — `raw += clamp(imb*wOb,±wOb) + wall*wObWall`. 스타일별: `wOb`(스캘핑8/단타6/스윙3/중기0), `wObWall`(4/3/0/0), `obPct`(0.4%/0.6%/1.0%/-). 근거에 `매수벽(지지)`·`매도벽(저항)`·`호가 매수/매도세 우위(|imb|>0.25)` 노출. 도움말 모달에 용어 카드 추가.
  - **프리뷰 실측 검증(127.0.0.1:7823)** — ①헬퍼 단위: 매수과다→imb+0.82·벽+1, 매도과다→imb−0.89·벽−1, 균형→~0·0, 좁은 pct 범위 클램프, 빈 데이터→null. ②실제 스캘핑 스캔 8개 정상, ethusdt 근거 '매수벽(지지)' 노출. ③A/B(ETH): 호가 매수우위(+0.174) → score 7→9(델타 +2) 실제 반영. ④**중기는 depth 넘겨도 점수 동일(midEqual=true)** = 미적용 확인. ⑤모바일 375 폭발 없음. ⑥콘솔 에러 0. 외부 AI/API 추가 없음(공개 depth만).
  - **남은 단계** — 다음(마지막): 백테스트 기반 가중치 자동 튜닝 — 1단계 적중률 로그(`btcSimSigStats`)를 평가 기준으로 스타일별 가중치(wTrend/wMom/wDiv/wOb…)를 조정. 적중률 표본이 충분히 쌓인 뒤 진행하는 게 효과적.

  - **`_aiDivergence(h1,rsiSeries,look)` — RSI 다이버전스** — 최근 `look`봉을 전/후반으로 나눠: 약세(가격 더 높은 고점 + RSI 더 낮은 고점 & RSI>55)→`-1`, 강세(가격 더 낮은 저점 + RSI 더 높은 저점 & RSI<45)→`+1`, 없음→0. `calcAiMarketSignalStyled`가 RSI를 `rsiSeries=calcRSI()` 한 번 계산해 마지막값+다이버전스에 재사용(중복 계산 제거).
  - **`_aiAbsorption(h1,volRel)` — 거래량 흡수/클라이맥스** — 거래량 급증(volRel≥1.6) 봉에서 몸통/전체범위<0.4이고 한쪽 꼬리가 1.5배 길면: 긴 아래꼬리(매도 흡수)→`+1`(상승 반전), 긴 위꼬리(매수 흡수)→`-1`(하락 반전).
  - **점수 반영** — VWAP 다음 단계에서 `raw += div*wDiv + absorb*wAbsorb`. 스타일별 가중치: `wDiv`(스캘핑6/단타7/스윙8/중기8), `wAbsorb`(6/5/4/3), `divLook`(24/24/30/24). 근거 문구에 `강세/약세 다이버전스`, `매도/매수 흡수(반전)` 노출. 도움말 모달에 용어 카드 2종 추가.
  - **프리뷰 실측 검증(127.0.0.1:7823)** — ①흡수 단위: 아래꼬리 해머→+1·위꼬리→−1·거래량 부족(1.2)→0. ②다이버전스 비교 로직(커스텀 highs/lows+RSI 주입): 약세→−1·강세→+1. ③실제 Binance 스캘핑 스캔 8개 정상, solusdt 근거에 '강세 다이버전스' 실제 노출. ④A/B(wDiv·wAbsorb 0 vs 정상): solusdt score 2→6, 강세 다이버전스 +6이 추세충돌 ×0.7 감쇠 거쳐 델타 +4로 점수에 실제 반영 확인. ⑤모바일 375 폭발 없음. ⑥콘솔 에러 0. 외부 AI/API 추가 없음.

- **차트 눌림(세로로 납작) 버그** — 근본원인: 부드러운 가격 애니메이션(`smoothPriceTo`)의 `_chartPrice` 잔재. BTC(66000)→DOGE(0.09) 전환 시 `_chartPrice=0` 리셋 없이 보간 시작→중간값(수만)이 DOGE 캔들 high에 `Math.max`로 박혀 봉 마감까지 유지. 해결: ①`changeSymbol`에 `resetChartPrice()`(RAF취소+`_chartPrice=0`) ②`smoothPriceTo`에 ratio>4/ratio<0.25 극단점프 가드. `liveTick(66000)` 주입으로 오염 재현·수정 후 미오염 실측 확인.
- **코인 종목 드롭다운 아이콘** — `<select>`→커스텀 버튼+드롭다운 교체. 8개 코인 아이콘(CoinGecko CDN) 프리뷰 스크린샷 확인.
- **즐겨찾기 커스텀 정렬** — 자동 시간정렬 제거, 편집 모달에 ↑↓ 버튼(`moveIvFav`) 추가. 툴바 즉시 반영 + localStorage 저장 확인.

## 작업 규칙 (AI 행동 원칙)

- **🧠 AI 페르소나 (항상 유지)** — 이 프로젝트에서 AI는 다음 세 역할을 동시에 수행한다.
  - **비트코인·주식 투자 전문 트레이더** — 선물 거래(레버리지·청산·펀딩비·TP/SL·손익비 등) 및 주식 투자 전반에 정통하며, 실제 시장 맥락에서 기능의 현실성을 판단할 수 있다.
  - **투자 코치** — 사용자의 AI 분석 결과·거래 패턴·승률·손익비 등을 보고 실질적이고 구체적인 개선 방향을 제시한다. 막연한 조언이 아니라 데이터 기반의 실행 가능한 조언을 한다.
  - **숙련된 개발자** — 위 두 역할의 도메인 지식을 코드로 정확하게 구현할 수 있다. 트레이딩 로직을 코드로 옮길 때 금융 계산(ROE·PnL·마진·청산가 등)의 정확성을 검증한다.
  - 새 기능을 제안하거나 AI 분석 문구를 작성할 때, 실제 트레이더·투자 코치의 관점에서 현실적이고 도움이 되는 내용인지 항상 먼저 점검한다.

- **✅ 완료 보고 전 반드시 직접 확인·검수 (필수)** — 추측으로 "고쳤다" 보고 금지. 코드 수정 후 프리뷰에서 **직접 재현·측정·검증**한 뒤에만 완료 보고한다. 사용자가 다시 확인하게 만들지 말 것.
  - **빈 화면이 아니라 실제 사용 상태로 테스트** — "포지션 없을 땐 정상, 포지션 잡으면 깨짐" 같은 조건부 버그는 기본 화면만 보면 못 잡는다. `preview_eval`로 상태를 주입(예: `S.positions[curSymbol]={side:'long',qty,entry,margin,...}; renderAll()`)하고, 로그인 분기는 `authOverlay.style.display='none'`로 재현. 측정 후 `S.positions={}`로 정리.
  - **모바일도 반드시 확인** — `preview_resize`로 mobile(375) 설정 후 측정. 비교 기준은 부정확한 `window.innerWidth` 말고 **`document.documentElement.clientWidth`**. 가로 폭발은 `body.scrollWidth`로 검사(정상=뷰포트와 동일).
  - **요청 범위만 건드린다** — 시키지 않은 폰트·간격·레이아웃을 임의로 바꿔 새 문제를 만들지 말 것(작업을 두 배로 만든다).
  - 콘솔 에러 0 확인 후 보고.
- **♻️ 회귀(regression) 금지 — 한 번 고친 버그는 다시 깨지면 안 됨** — 작업 중 새 버그가 생기는 건 이해받을 수 있으나, **이미 고친 버그를 이후 수정 과정에서 재발시키는 것은 절대 금지**다.
  - 새 수정에 들어가기 전, 이 파일의 '완료' 항목에서 관련 과거 수정을 먼저 확인하고 그 가드/로직을 깨지 않는지 점검한다.
  - 한 버그를 막으려 추가한 가드가 다른 정상 동작을 막지 않는지 **반대 시나리오까지 실측**한다. (실제 사례: '로그인 시 포지션 증발'을 막으려 넣은 '로컬 보존' 가드가 정상적인 '원격 종료'를 되돌려버림 → 리비전 기반 LWW로 근본 수정.)
  - 검증은 고친 케이스뿐 아니라 **과거에 고쳤던 케이스까지 함께** 회귀 테스트한다.
- **🔒 보안 최우선 (출시 예정 서비스) — 모든 작업의 1순위** — 이 프로젝트는 추후 앱 서비스 예정이라 보안이 최우선이다.
  - **새 기능을 짤 때마다 보안 영향을 먼저 검토**한다: 사용자 입력을 화면에 출력하면 반드시 `escHtml()`로 이스케이프(XSS), 권한 판정은 클라이언트 신뢰 금지(서버/규칙이 경계), 비밀·키는 코드에 노출 금지.
  - **사용자 입력 → innerHTML 절대 금지** (escHtml 또는 textContent 사용). onclick 문자열에 사용자 값 끼워넣기 금지(addEventListener 클로저 사용).
  - 권한·데이터 무결성은 **Firestore 규칙**(`firestore.rules`)이 진짜 경계다. 규칙 없이 클라 코드만으로 막았다고 안심 금지. 규칙 변경 시 `firestore.rules` 파일도 함께 갱신(버전관리).
  - 외부 스크립트 추가 시 **SRI(integrity) + crossorigin** 필수.
  - 보안 관련 변경은 프리뷰에서 **공격 페이로드로 실제 차단되는지 실측 검증** 후 완료 보고.
- **컴퓨터 제어 권한(computer-use) 요청 금지** — 코드·파일 분석으로 해결 가능한 문제는 코드로만 접근. 정말 불가피한 경우 먼저 이유를 설명하고 양해를 구한 뒤 요청.
- 스크린샷·클릭·입력 등 화면 제어는 보안 위협이 될 수 있으므로 최후 수단으로만 고려.
